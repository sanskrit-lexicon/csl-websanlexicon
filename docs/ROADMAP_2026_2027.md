# csl-websanlexicon — Security & Vulnerability-Avoidance Roadmap 2026–2027

_Created: 03-07-2026 · Last updated: 14-07-2026_

Scope: hardening the **CDSL web-frontend** — the Python + Mako generator in
[`v02/`](https://github.com/sanskrit-lexicon/csl-websanlexicon/tree/main/v02) that renders
the per-dictionary PHP + SQLite web apps deployed to
`sanskrit-lexicon.uni-koeln.de`. This roadmap covers **security posture only**;
display/content correctness lives in the GitHub issue tracker, and dictionary
*content* fixes belong in [`csl-orig`](https://github.com/sanskrit-lexicon/csl-orig).

Grounded in the **2026-07-03 security review** (5 code findings) plus a repo-settings
audit (GitHub security features, SAST gating, branch protection). Interview rulings
are recorded under [Decisions taken](#decisions-taken).

## Wave 2 — Make SAST gate regressions (Q3 2026) — ✅ DONE (03-07-2026)

**Goal:** a new PR that introduces a security finding cannot be merged silently.

Deliverables:
- Convert Semgrep from advisory (`|| true`, SARIF-only) to a **blocking check on new
  findings** — either GitHub Code Scanning's PR-diff gate, or a `semgrep ci` step that
  fails on new results while grandfathering the existing baseline.
- Turn on **branch protection** for `main` with the SAST check + the existing lint checks
  (`PHP syntax lint`, `Python lint + Mako compile`, `YAML lint`, `Analyze (python)`) as
  **required status checks**, require a PR before merge, and dismiss-stale-approvals.
- Keep CodeQL (Python) advisory — it has no PHP analyzer, so Semgrep is the real gate for
  the ~800-file PHP tree.

**Delivered:** [`semgrep-ci-gate`](https://github.com/sanskrit-lexicon/csl-websanlexicon/blob/main/.github/workflows/semgrep.yml)
job running `semgrep ci` (diff-aware) — [PR #77](https://github.com/sanskrit-lexicon/csl-websanlexicon/pull/77),
merged, v0.2.1. Verified on a throwaway PR (#79) that an injected finding fails the gate and a
no-op PR passes.

**Unblocked by:** Wave 1 (baseline clean, so gating won't wall off an existing backlog).
**Branch protection ✅ APPLIED 03-07-2026** (Opus 4.8 `claude-opus-4-8`, via `gh api` admin
token): `main` now enforces all 5 required status checks (`Semgrep CI gate (new findings)`,
`PHP syntax lint (templates)`, `Python lint + Mako compile`, `YAML lint`, `Analyze (python)`) +
require-PR-before-merge + dismiss-stale-approvals. `required_approving_review_count` kept at 0
(preserves hands-off auto-merge); advisory `Semgrep SAST` upload job left non-required. The gate
is now genuinely blocking.

**Ruling:** [D3](#decisions-taken) — *make Semgrep a required check on new alerts.*

## Wave 3 — Security headers + phased CSP in the generated apps (Q4 2026) — 🟡 STAGE 1+2 DONE (14-07-2026)

**Goal:** every generated dictionary app ships defensive HTTP headers, independent of
whatever the Cologne Apache config does.

Approach (per ruling [D4](#decisions-taken) — *headers in templates, phased CSP*):

1. **Shared header include** — a `security_headers.php` in the template tree that emits, on
   every request, before any output:
   - `X-Content-Type-Options: nosniff`
   - `Referrer-Policy: strict-origin-when-cross-origin`
   - `X-Frame-Options: SAMEORIGIN` (the apps are not meant to be framed cross-site)
   `require_once` it at the top of each HTML-emitting entry point (`indexcaller*.php`,
   `webtc1/*`, `webtc2/query.php`, `mobile1/index.php`, `getword.php`, `servepdf*`).
2. **CSP in Report-Only first** — add `Content-Security-Policy-Report-Only` to surface every
   inline-script / inline-style / external-origin violation *without breaking the page*. The
   apps use inline `<script>` blocks and jQuery, so a naive enforcing CSP would break them —
   Report-Only is the measurement pass.
3. **Tighten, then enforce** — from the Report-Only telemetry, hash/nonce the required inline
   scripts, pin the allowed origins (self + the corrections host), then flip to enforcing
   `Content-Security-Policy`.

Because headers live in the template, they travel with every `generate.py` regeneration and
survive server-config drift.

**Delivered (stage 1+2, 14-07-2026):** [`v02/makotemplates/web/security_headers.php`](https://github.com/sanskrit-lexicon/csl-websanlexicon/blob/main/v02/makotemplates/web/security_headers.php)
emits the three baseline headers plus `Content-Security-Policy-Report-Only`, `require_once`'d
at the top of the 9 real HTML/JSON-emitting entry points (before any other code/output):
`webtc/indexcaller.php`, `webtc/indexcaller_fr.php`, `webtc/getword.php`, `webtc/servepdf.php`,
`webtc1/index.php`, `webtc1/disphier.php`, `webtc1/listhier.php`, `webtc2/query.php`,
`mobile1/index.php`. The `webtc1` support/class files (`listparm.php`, `listhiermodel.php`,
`listhierview.php`) were deliberately **not** given the require — they are `require_once`'d
*after* their caller already echoes output, so adding a header call there would warn
"headers already sent" instead of helping.

CSP choice: the Report-Only policy does **not** include `'unsafe-inline'` on
`script-src`/`style-src` (unlike the "start permissive" option sketched above) — with
`'unsafe-inline'` present the browser silently allows all inline script/style and reports
nothing, which would defeat the stated goal of *surfacing* inline-script/inline-style
violations for the stage-3 hash/nonce migration. Known-legitimate external origins (the
jQuery Mobile CDN `code.jquery.com` used by `mobile1/`, the `www.sanskrit-lexicon.uni-koeln.de`
scans host used by `servepdf.php`'s PDF/image embed) are allowlisted so only genuinely
unexpected external-origin loads show up as report noise. **A human should decide** whether
this stricter-than-sketched Report-Only policy is preferred over the more permissive
`'unsafe-inline'` variant before Wave 3 stage 3 (enforce) is scheduled — see the exact policy
string in `security_headers.php`.

**Stage 3 (tighten + enforce) is explicitly NOT implemented** — it requires real Report-Only
violation telemetry from a live Cologne deployment (this session cannot generate that in one
sitting) to know which inline scripts to hash/nonce and which origins are genuinely needed.
Follow-up: once the CDSL maintainers regenerate + deploy this template change, collect
Report-Only console/telemetry data for a representative sample of dictionaries, then open a
fresh handoff for stage 3.

**Unblocked by:** Wave 1 (entry-point inventory already mapped in the review).
**Coordination:** regeneration + deployment is done by the CDSL maintainers (Jim/Dhaval) —
the template change is ours, the deploy is theirs. Flag on the regeneration handoff.

## Wave 4 — Reduce attack surface & dependency risk (Q1 2027)

- **Delete `v00/`** (currently excluded-not-deleted per D2, revisit once #72 lands): once
  the server is confirmed to serve only `v02/` paths and issue
  [#72](https://github.com/sanskrit-lexicon/csl-websanlexicon/issues/72) (*removing connection
  with old-websanlexicon*) is resolved, drop the dead tree entirely. Git history preserves it.
- **JS dependency audit** — the apps vendor `jquery.min.js`, `jquery.cookie.js`,
  `orphus.customized.js`. Inventory versions, check for known CVEs, and pin/refresh. These
  are not covered by Dependabot (no manifest) — a manual `retire.js`-style pass.
- **PHP floor** — record the minimum supported PHP version and confirm the escaping/`json_encode`
  flags used in Wave 1/3 behave identically across the server's PHP (the `abot` display code
  was recently made "work across all PHP versions", [#61](https://github.com/sanskrit-lexicon/csl-websanlexicon/issues/61)).

**Unblocked by:** Wave 3 (CSP telemetry may name the risky vendored scripts) and issue #72.

## Decisions taken

Rulings elicited from M.G. in the 03-07-2026 roadmap interview (options + recommendation
presented; the chosen option is recorded verbatim below):

- **D1 — Handle the five confirmed findings vs. roadmap.** → **Fix all five now + roadmap
  posture only.** Rationale: two were live HIGH reflected-XSS/traversal vectors on a public
  university origin; they should not wait for a future session to execute a roadmap. Posture
  work (SAST gating, headers) is scheduled, not rushed. → Wave 1, [PR #76](https://github.com/sanskrit-lexicon/csl-websanlexicon/pull/76).
- **D2 — The dead `v00/` legacy tree.** → **Keep but exclude from scanners.** Rationale:
  retain the 2020 tree as reference (per the readme) but stop it raising alerts; full deletion
  waits on confirming the server never serves `v00/` paths (Wave 4, gated on #72). → Semgrep
  `--exclude v00` + CodeQL `paths-ignore`.
- **D3 — SAST gating.** → **Make Semgrep a required check on new alerts.** Rationale: block
  regressions without a big backlog-fix burden by grandfathering the (now-clean) baseline;
  small, acceptable maintainer friction. → Wave 2.
- **D4 — Where hardening HTTP headers live.** → **Add headers in templates, phased CSP
  (Report-Only → enforce).** Rationale: template-level headers travel with every regeneration
  regardless of the Cologne server config, which this repo does not control; Report-Only first
  avoids breaking the apps' inline scripts. → Wave 3.

## Non-goals (considered and ruled out — do not re-propose)

- **Server-level (Apache/nginx) header config as the primary mechanism.** Weighed in D4 and
  set aside: this repo does not own the uni-koeln.de server config, so a server-only approach
  is out of our control and needs maintainer buy-in. Templates are the lever we own. (A
  server-level *belt-and-suspenders* layer is welcome if the maintainers add it, but it is not
  a deliverable here.)
- **Deleting `v00/` immediately.** Ruled out in D2 in favour of scanner exclusion until the
  server-path confirmation (Wave 4).
- **Rewriting the PHP frontend / adopting a framework.** Out of scope — the generator + flat
  PHP is the maintained architecture; security work hardens it in place.
- **Dictionary content / display-correctness fixes.** Tracked as GitHub issues (content →
  `csl-orig`); not security, not this roadmap.
- **Blocking (non-advisory) CodeQL.** CodeQL has no PHP analyzer; Semgrep is the real gate.
  CodeQL stays advisory for the Python generator only.
